### 1.1 
### Foundational Models

Foundation models are large language models trained on massive datasets that can perform a wide range of tasks such as text generation, summarization, question answering, and reasoning. In the LangChain framework, these models are accessed through a unified interface that allows developers to interact with different providers such as OpenAI, Anthropic, or Google.

In the notebook 1.1_foundational_models.ipynb, the main objective is to demonstrate how to initialize and interact with a language model. The model is first created using the init_chat_model() function, which allows the user to specify the desired model and configuration parameters such as temperature. Once initialized, the model can be invoked using the .invoke() method, which sends a prompt to the model and returns a structured response object.

The response returned by the model is typically an AIMessage object, which contains the generated content as well as additional metadata such as token usage or response information. Extracting the actual generated text can be done by accessing the content attribute of the response.

Another important aspect covered in this notebook is model configuration. Parameters such as temperature allow developers to control the creativity and variability of the model’s responses.

### Prompting

The notebook 1.1_prompting.ipynb introduces the concept of prompting, which refers to the design of instructions given to the language model in order to influence its behavior and output. Prompting is a fundamental concept when working with large language models, as the quality and structure of the prompt significantly affect the results produced by the model.

One important technique is the use of system prompts, which define the role or behavior that the model should adopt. For example, the model can be instructed to behave as a scientist, teacher, or science fiction writer. This allows developers to guide the style and content of the responses generated by the model.

Another technique demonstrated in the notebook is few-shot prompting, where examples of desired inputs and outputs are provided to the model. By observing these examples, the model can infer the expected structure of the response and generate outputs that follow a similar pattern.

Finally, the notebook introduces structured outputs, which allow the responses of the model to be constrained to a predefined schema. This approach is particularly useful in real-world applications, as it enables the integration of language model outputs into software systems in a reliable and structured way.

## 1.2 - Tools
### Tools


In agentic AI systems, tools play a fundamental role in extending the capabilities of large language models (LLMs). While LLMs are powerful for reasoning and natural language understanding, they are limited when it comes to performing precise computations, accessing external data sources, or interacting with software systems. Tools address these limitations by allowing the model to call external functions implemented in code.

In the LangChain framework, a tool is typically defined as a Python function that performs a specific task. By decorating a function with the `@tool` decorator, the function becomes accessible to an AI agent. This decorator registers the function as a callable tool and exposes its name, parameters, and description to the language model. The description is particularly important because it helps the model decide when the tool should be used in response to a user query.

Tools can also be invoked directly through the LangChain interface using the `invoke()` method. This allows developers to test and validate the functionality of the tool independently of the agent. The method typically receives input parameters in dictionary form and returns the computed result.

Once tools are defined, they can be provided to an AI agent during its creation. In LangChain, an agent is constructed by specifying the underlying language model, the list of available tools, and a system prompt that defines the behavior of the agent. When the agent receives a user query, the language model analyzes the request and decides whether one of the available tools should be used to obtain a more accurate or reliable result.

In the context of the AI4MDR project, the concept of tools becomes particularly relevant. The agent can be equipped with tools designed to interact with regulatory documentation and software development artefacts related to medical device certification. For instance, tools could be created to retrieve specific articles from the MDR regulation, extract relevant requirements from technical documentation, or generate traceability links between regulatory requirements and software development artefacts. By combining these specialized tools with the reasoning capabilities of an LLM, it becomes possible to build an agent capable of assisting in the generation and verification of certification evidence within the medical device regulatory workflow.

## Lesson 1 - What is Middleware?

Usar um agente normal funciona assim:
User-> Modelo -> Tool -> Modelo -> Resposta
O problmea deste funcionamento é que não temos controlo, não conseguimos impor regras facilmente e não é seguro para pordução

*Middleware* são funções que metemos no meio deste loop para controlar o que acontece
No caso do AI4MDR, o Middleware pode verificar se resposta tem fontes, bloquear respostas sem evidência, e obrigar a citar nomes
Exemplo:
if resposta_nao_tem_referencia:
    bloquear_resposta()

## Lesson 2 - Human in the loop
Nesta lesson aprende-se a gerir um porblema fundamental dos agentes - como gerir conversas longas se perder performance, qualidade e controlo.

Problema que está a ser resolvido: os agentes guardam o histórico com+leto da conversa no state (memória). Com o tempo o agente começa a degradra-se à edida que a conversa cresce.

A dsolução passa por *controlar o histórico da conversa usando middleware*, permitindo intervir no fluxo do agente emodificar o estado antes ou depois de cada execução.

Vamos usar duas estratégias:
 *Summarization ( resumir a conversa)*
- Em vez de guardar todas as mensagens o agente cria um resumo das interações anteriores
- mantém apenas esse resumo + as mensagens mais recentes

*Trimming (cortar mensagens)
Em vez de resumir:
- remove dretamente mensagens do histórico
- Pode ser feito de várias formas:
  - apagar as mensagens mais antigas
  - removermensagens irrelevantes 

## Lesson 3: Human in the loop
A lesson 3 é na prática, uma introdução a Human _ in_the_Loop (HITL) em Langchain: o agente lê o estado, decide usar ferramentas, e  o middleware pode interromper a execução antes de certas tools correrem, para um humano aporvar,rejeitarm ou editar.

## Lesson 4: Dynamic Agents

Nesta parte vamos fazer agentes que mudam ocomportamento dinamicamente, isto é, se estamos perante a situação X -> faz X; se estamos perante situação B -> faz Y

AGentes simples funcionam bem como MVP, mas em produção é necessário adaptar o comportamento do agente consoante:
- tipo de utilizador
- contexto
- complexidade da tarefa

É introduzido um middleware como forma de controlar o comportamento do agente.

*1.Dynamic Prompts*
Permite mudar o prompt em runtime.
Exemplo dado:
- responder em inglês
- responder na língua do utilizador

*Dynamic Tools*
Permite controlar que tools o agente pode usar
Exemplo:
- utilizador interno -> acesso a base de dados + web
- utilizador externo -> apenas web

*Dynamic Models*
Permite trocar o modelo durante a execução
Exemplo dado:
- conversa curta -> modelo leve
- conversa longa -> modelo mais avançado
